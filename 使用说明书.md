# NovelCraft Pro 使用说明书

当前版本基线：Phase 11（Release Hardening / Pre-Ship Cleanup Minimal Chain）

本说明书面向第一次接手仓库、准备本地运行、演示、验收或交付的使用者，重点说明：

- 如何启动服务
- 如何导入示例项目包
- 如何通过 `/studio` 完成最小工作流
- 如何使用导出 / 导入 / 归档 / 备份 / 重建 / 诊断
- 当前仓库中 scene / chapter / volume / book 主链的使用方式

本版本已经具备可演示、可导出、可导入、可备份、可归档、可体检的完整最小链，但仍坚持 deterministic / rule-based 优先，不扩展新的智能写作模块。

## 1. 项目简介

NovelCraft Pro 是一个面向中文长篇网文创作的人机协同系统。当前主线已经覆盖：

- consultant 立项问答
- bible / planner 结构化规划
- scene draft 生成、checks、accept / reject、repair
- accepted scene 驱动的 memory 与 retrieval
- chapter / volume / book assembled / finalized
- book continuity checks
- 成品化：
  - 导出
  - 重建
  - 诊断 / 体检
  - 项目包
  - 导入项目包
  - 归档快照
  - 备份
  - 快照列表
- Phase 11 发布前收口：
  - 示例项目包
  - 演示流程
  - 发布检查清单
  - 发布前只读检查

当前设计原则：

- deterministic / rule-based 优先
- 不引入 LLM critic
- 不做 book-level repair / rewrite / polish
- `/studio` 以同步工作台为主，不使用任务队列或后台任务系统

## 9. 核心写作主链

本项目当前可用的完整最小主链如下。

### 9.1 立项与规划

顺序通常是：

1. 创建项目
2. 启动顾问会话
3. 回答顾问问题
4. 完成立项定稿
5. 生成 Bible 文档
6. 确认各类 Bible 文档
7. 生成并确认规划文档

典型 API：

- `POST /api/projects`
- `POST /api/projects/{project_id}/consultant/sessions`
- `POST /api/consultant/sessions/{session_id}/answer`
- `POST /api/consultant/sessions/{session_id}/finalize`
- `POST /api/projects/{project_id}/bible/from-consultant`
- `POST /api/projects/{project_id}/characters/confirm`
- `POST /api/projects/{project_id}/bible/world/confirm`
- `POST /api/projects/{project_id}/bible/power-system/confirm`
- `POST /api/projects/{project_id}/bible/story-bible/confirm`
- `POST /api/projects/{project_id}/plans/volumes/generate`
- `POST /api/projects/{project_id}/plans/master-outline/confirm`
- `POST /api/projects/{project_id}/plans/volumes/{volume_id}/confirm`
- `POST /api/projects/{project_id}/plans/volumes/{volume_id}/chapters/generate`
- `POST /api/projects/{project_id}/plans/chapters/{chapter_id}/confirm`
- `POST /api/projects/{project_id}/plans/chapters/{chapter_id}/scenes/generate`
- `POST /api/projects/{project_id}/plans/scenes/{scene_id}/confirm`

### 9.2 场景草稿

场景是实际写作主链的起点：

- `POST /api/projects/{project_id}/drafts/scenes/{scene_id}/generate`
- `POST /api/projects/{project_id}/drafts/{draft_id}/repair`
- `POST /api/projects/{project_id}/drafts/{draft_id}/accept`
- `POST /api/projects/{project_id}/drafts/{draft_id}/reject`

当前策略：

- checks 优先 deterministic / rule-based
- 检索已可消费已接受场景、章节摘要、卷摘要、整书摘要
- 修复只做 targeted repair，不扩展为整书级 rewrite

### 9.3 章节 / 卷 / 整书

章节：

- `POST /api/projects/{project_id}/chapters/{chapter_id}/assemble`
- `POST /api/projects/{project_id}/chapters/{chapter_id}/finalize`

卷：

- `POST /api/projects/{project_id}/volumes/{volume_id}/assemble`
- `POST /api/projects/{project_id}/volumes/{volume_id}/finalize`

整书：

- `POST /api/projects/{project_id}/book/assemble`
- `POST /api/projects/{project_id}/book/finalize`
- `GET /api/projects/{project_id}/book/continuity-checks/latest`
- `POST /api/projects/{project_id}/book/continuity-checks/recheck`

说明：

- assembled / finalized 都是派生层，不是 canonical 真相源
- continuity checks 是整书定稿前的整书级 deterministic 预检查

## 10. 成品化能力

Phase 10A / 10B / 11 合并后，成品化能力已形成完整最小链。

### 10.1 导出

接口：

- `POST /api/projects/{project_id}/export`

支持范围：

- `scene`
- `chapter`
- `volume`
- `book`
- `project`

支持 format：

- `markdown_package`
- `json_package`

说明：

- `scene` 导出要求唯一 active accepted draft
- `chapter / volume / book` 允许导出当前 `assembled / finalized / stale` artifact
- `project` 导出是完整 project package
- export 是只读快照，不会隐式触发 assemble / finalize / rebuild / recheck

### 10.2 重建

接口：

- `POST /api/projects/{project_id}/rebuild`

默认全量顺序：

1. `memory`
2. `chapters`
3. `volumes`
4. `book`
5. `checks`

原则：

- 重建必须幂等
- 不自动提升到 finalized
- 只重建缺失派生层
- 快照仍匹配的 finalized 保留
- 否则降为 `stale`

### 10.3 诊断 / 体检

接口：

- `GET /api/projects/{project_id}/diagnostics/health`

特点：

- 严格只读
- 不触发 rebuild
- 不触发 recheck
- 不更新 memory
- 不刷新 artifact 状态

适用场景：

- 工作台加载项目信息
- 导入后做只读校验
- 发布前检查项目状态

### 10.4 导入项目包

接口：

- `POST /api/projects/import-package`

MVP 限制：

- 只支持 `create_new`
- 不允许覆盖已有项目目录
- `project_id / slug` 已存在时直接 `409`
- 导入逻辑以 `canonical / derived / memory / checks` 为核心
- `markdown_package` 不是恢复核心依赖

### 10.5 Archive / Backup / Snapshots

接口：

- `POST /api/projects/{project_id}/archive-snapshot`
- `POST /api/projects/{project_id}/backup`
- `GET /api/projects/{project_id}/snapshots`

约定：

- archive 可带 `label`
- backup 默认无 label
- snapshots 默认按 `created_at desc` 返回
- archive 与 backup 共用最小列表

## 12. 错误与边界说明

当前错误边界收口主要覆盖成品化 / 工作台直连端点。

常见 HTTP 语义：

- `400`
  - 请求参数非法
  - package manifest / inventory 不合法
- `404`
  - 项目不存在
  - 目标资源不存在
- `409`
  - scene export 没有唯一 accepted draft
  - 目标 project_id / slug 已存在
  - 目标 artifact 当前不可导出
- `422`
  - 请求体结构不合法

工作台对成品化错误的展示是：

- `status`
- `code`
- `message`

不会显示 Python 堆栈，也不会自动重试。

## 13. 运行时目录与产物说明

项目运行过程中，常见输出包括：

- `projects/<project_id>/...`
  - canonical
  - drafts
  - memory
  - checks
  - exports
  - archives
  - backups

其中：

- canonical 是项目结构化真相源
- assembled / finalized / memory / checks 都是派生层
- rebuild 用于重建可稳定推导的派生状态
- archive / backup 用于保存历史时间点快照
- import 用于从 package 恢复一个新项目副本
